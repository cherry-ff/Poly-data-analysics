# Polymarket BTC 15min 机会挖掘

## 1. 先说结论

基于当前 `runtime_data` 里的真实数据结构，我认为现在最值得做的，不是继续抽象地“优化 fair”，而是把机会挖掘分成下面四类，并按优先级推进：

1. `full-set` 错价
2. `fast_close` 阶段的终局收敛
3. `ACTIVE` 阶段的方向性 taker edge，但必须加 realization filter
4. maker quote 质量分析，而不是直接拿 raw edge 下单

核心原因很简单：

- 你现在存的不只是 Polymarket 盘口
- 你还存了 fair/theo、quote plan、market lifecycle、global 基准源
- 但你没有稳定可用的“真实成交回执 / fill / cancel / ack”数据

所以当前最稳妥、最有信息量的工作流应该是：

```text
先做可成交回放
→ 再把盈利拆成 fair drift 和 convergence
→ 再按 phase / regime / liquidity 分层
→ 最后只保留历史上真的能兑现的结构
```

## 2. 当前 `runtime_data` 里实际存了什么

先把真实数据资产讲清楚。当前不是“我有一些 BTC 价格数据”，而是已经有一套比较完整的事件流。

### 2.1 全局数据 `runtime_data/records/global`

当前确认有这些 topic：

- `feeds_binance_tick`
- `feeds_binance_depth`
- `feeds_chainlink_tick`
- `feeds_polymarket_market_book_top`
- `feeds_polymarket_market_depth`
- `feeds_polymarket_market_new_market`

其中最关键的是：

- Binance tick: `last_price / best_bid / best_ask`
- Binance depth: 5 档盘口
- Chainlink tick: `price / bid / ask / oracle_ts_ms`
- 全局 Polymarket new_market: 新 market 发现时刻

这意味着你现在的外部基准源，真实可用的是：

- Binance 短时价格与盘口
- Chainlink 慢一点但更“锚”的价格
- Polymarket 全局发现流

不是文档里原来写的 OKX。

### 2.2 单 market 数据 `runtime_data/records/markets/<market_id>`

当前确认有这些 topic：

- `market_metadata`
- `market_lifecycle_transition`
- `feeds_polymarket_market_book_top`
- `feeds_polymarket_market_depth`
- `pricing_theo`
- `pricing_quote_plan`
- 部分旧市场有 `strategy_order_intents`
- 已结束市场会有 `feeds_polymarket_market_resolved`

这些字段足够支持下面几件事：

- 用 `book_top/depth` 做可成交价回放
- 用 `pricing_theo` 看 fair edge 是否兑现
- 用 `market_lifecycle_transition` 做 `PREWARM / ACTIVE / FAST_CLOSE` 分层
- 用 `pricing_quote_plan` 看策略当时想怎么报 maker quote
- 用 `resolved` 做终局相关分析

### 2.3 一个重要现实：当前缺什么

当前我没有在 `runtime_data` 里看到稳定的：

- 成交回执
- 撤单回执
- 订单 ack
- fill 明细

这意味着：

- 你现在最靠谱的是“模拟可成交回放”
- 还不能把历史收益写成“真实成交收益”
- maker 策略现在更适合先做“理论质量分析”，不适合直接声称真实成交表现

另外，最近 250 个市场里，`strategy_order_intents` 基本没有继续落盘；但 `pricing_quote_plan` 仍然持续存在，且最近 50 个市场的 quote reason 主要是：

- `maker_active`
- `maker_fast_close`
- `maker_prewarm`
- `maker_discovered`

这说明当前系统仍然在做 phase-aware quote planning，只是最近不再稳定记录具体 intent。

## 3. 基于当前数据，哪些问题可以立刻回答

### 3.1 可以立刻做的

1. `taker edge` 是否真的在未来 30s / 60s / 120s 内兑现
2. 收益主要来自 `fair drift` 还是 `edge convergence`
3. 哪些 phase 最容易出现假 edge
4. `sum(best ask)` 或 `sum(best bid)` 是否存在 full-set 错价
5. maker quote 在不同 phase 下是否明显更容易被 adverse selection

### 3.2 现在不适合直接做结论的

1. “真实成交后净收益是多少”
2. “maker 实盘 fill rate / fill quality 到底如何”
3. “某条 intent 最终是否真的上链成交”

这些都需要补 execution log。

## 4. 我用现有数据做的一个简化样本回放

我先用当前 `runtime_data` 做了一个轻量实证，样本是：

- 最近 250 个 market
- 其中 240 个 market 具备可用的 `pricing_theo + pair book`
- 不含手续费、滑点和未成交惩罚
- 用 visible top-of-book 做模拟 taker 回放

也就是说，这个结果对 raw edge 已经是偏乐观的；如果它都不赚钱，真实结果通常只会更差。

### 4.1 方向性 raw fair edge 直接做 taker，整体是负的

定义：

```text
edge_yes(t) = theo_up(t) - ask_yes(t)
edge_no(t)  = theo_down(t) - ask_no(t)
```

模拟收益：

```text
Π_yes(t, H) = bid_yes(t+H) - ask_yes(t) - cost_buffer
Π_no(t, H)  = bid_no(t+H)  - ask_no(t)  - cost_buffer
```

在这次样本里：

- `ACTIVE` 段 raw edge 直接做，YES 和 NO 在 30s / 60s / 120s 上平均收益都为负
- 中位数基本都是 `-0.01`
- 即使 edge 阈值提到 `0.03`，平均收益仍然大多为负

一个典型结果是：

- YES，`edge > 0.02`，持有 60s，平均收益约 `-0.0044`
- NO，`edge > 0.02`，持有 60s，平均收益约 `-0.0051`

注意这还是没扣交易成本的结果。

### 4.2 负收益不是因为没有收敛，而是因为 fair drift 在打脸

把收益拆成：

```text
Π = fair_gain + convergence_gain - cost
```

这次样本里，一个非常清楚的现象是：

- `convergence_gain` 平均是正的
- 但 `fair_gain` 平均更负
- 所以总 PnL 仍然为负

这说明你之前的直觉是对的：

> edge 看起来会收，但 fair 自己后面也会反着走，导致你最终赚不到。

换句话说：

- 问题不只是“poly 不收敛”
- 更关键的是“你抓到的 edge 常常伴随着后续 fair 反转”

### 4.3 `PREWARM` 很差，`FAST_CLOSE` 反而明显更好

在 `edge > 0.02`、持有 60s 的简化回放下：

- `PREWARM` 的 YES / NO 都明显差
- `ACTIVE` 仍然整体负
- `FAST_CLOSE` 的平均收益反而显著转正

样本结果大致是：

- YES `FAST_CLOSE`: 平均约 `+0.051`
- NO `FAST_CLOSE`: 平均约 `+0.073`

这说明：

- `FAST_CLOSE` 不是普通 edge 逻辑
- 它更像终局阶段的特殊收敛 / stale quote / 清算前补价结构
- 应该单独建模，不能和普通 ACTIVE edge 混在一起

### 4.4 `full-set` 错价确实存在，但不高频

定义：

```text
fullset_buy_edge(t) = 1 - ask_yes(t) - ask_no(t)
fullset_sell_edge(t) = bid_yes(t) + bid_no(t) - 1
```

在最近 250 个 market 的样本中：

- `ask_yes + ask_no < 0.99` 出现了 26 次，分布在 20 个市场
- `bid_yes + bid_no > 1.01` 出现了 30 次，分布在 26 个市场

按 phase 看：

- `ask_sum < 0.99` 主要出现在 `ACTIVE`，少量在 `FAST_CLOSE`
- `PREWARM` 基本没有

折价幅度也不小：

- `ask_sum < 0.99` 的平均折价约 `2.5%`
- 最大折价约 `7%`

这类机会不高频，但从结构上更接近真错价，因为它不依赖 fair 方向判断。

## 5. 当前最值得挖的机会，应该怎么排序

### 5.1 第一优先级：`full-set` 错价

这是当前最干净的一类。

原因：

- 不需要判断 BTC 下一步方向
- 不依赖 `theo_up / theo_down` 是否足够准
- 直接利用二元市场两边价格和的失衡
- 结构上更接近套利，而不是预测

应重点盯：

- `ask_yes + ask_no < 1 - cost_buffer`
- `bid_yes + bid_no > 1 + cost_buffer`

这里的 `cost_buffer` 不能省略，至少要覆盖：

- taker fee
- 滑点
- 盘口瞬时失真
- 下单延迟

### 5.2 第二优先级：`FAST_CLOSE` 特殊收敛

这类点不能和普通 ACTIVE 策略混用，但值得单独挖。

要点：

- phase 单独分层
- 按距结算剩余秒数切 bucket
- 看 book 是否经常出现 stale / one-sided / 清算前补价
- 把它当“终局阶段微结构策略”，不要再叫普通 fair edge

### 5.3 第三优先级：ACTIVE 阶段的 `raw edge + realization filter`

这类不是不能做，而是不能再按：

```text
fair → edge → 直接下单
```

必须升级为：

```text
fair
→ 候选 edge
→ realization filter
→ execution filter
→ trade
```

因为当前样本已经说明：

- raw edge 有收敛
- 但 fair drift 经常更糟
- 所以你必须先预测“这个 edge 会不会在未来窗口里兑现”

### 5.4 第四优先级：maker quote 质量分析

当前 `pricing_quote_plan` 很完整，可以做这些：

- quote 离市场 top 有多远
- quote 离 theo 有多远
- 不同 reason 下 quote 的后续 adverse selection 如何
- `maker_active / maker_fast_close / maker_prewarm` 哪类更安全

但因为最近缺真实 intent/fill，不要直接把它写成“maker 策略收益分析”。

### 5.5 当前最不该优先做的：`PREWARM` raw edge

原因已经被样本打脸：

- 兑现率差
- 平均收益差
- 很多 edge 是 discovery / 开盘前流动性不稳造成的假象

所以 PREWARM 更适合：

- 做 market discovery latency 分析
- 做 quote readiness 分析
- 做 opening regime 特征统计

不适合直接拿 raw edge 做 taker。

## 6. 现在应该怎样定义“可回放的交易”

由于没有真实 fills，当前统一定义成“模拟可成交交易”。

### 6.1 方向性 taker 回放

做多 YES：

```text
entry = best_ask_yes(t)
exit  = best_bid_yes(t+H) 或 barrier exit
Π_yes = exit - entry - cost_buffer
```

做多 NO：

```text
entry = best_ask_no(t)
exit  = best_bid_no(t+H) 或 barrier exit
Π_no  = exit - entry - cost_buffer
```

### 6.2 full-set 回放

买 full-set：

```text
entry = ask_yes(t) + ask_no(t)
exit  = 1.0
Π_fullset_buy = 1 - entry - cost_buffer
```

如果有能力做另一侧，也可以定义：

```text
Π_fullset_sell = bid_yes(t) + bid_no(t) - 1 - cost_buffer
```

但这部分要考虑你的实际执行约束。

### 6.3 maker 理论质量回放

这里不直接算真实收益，而是先算：

- quote 后未来 5s / 10s / 30s 是否被 adverse move
- 当时 quote 是否离市场太远
- 当时 quote 是否离 theo 太近但处于高风险 regime

## 7. 事件标签应该怎么做

当前建议至少做三套标签。

### 7.1 方向性 taker 标签

```text
y_pnl = 1 if Π > 0 else 0
```

并同时记录：

- `fair_gain`
- `convergence_gain`
- `capture_ratio`
- `MFE / MAE`

### 7.2 pure convergence 标签

```text
conv_gain > α
|fair_gain| < β
Π > 0
```

这类最接近你真正想要的“补价 alpha”。

### 7.3 full-set 错价标签

```text
fullset_buy_good = 1 if ask_yes + ask_no < 1 - cost_buffer
fullset_sell_good = 1 if bid_yes + bid_no > 1 + cost_buffer
```

这类标签不需要依赖 fair。

## 9. 当前最小闭环，应该怎么做

### 实验 1：做一张统一事件回放表

按事件采样，不要每条 tick 都打点。

建议候选事件：

- `edge_yes` 首次超过阈值
- `edge_no` 首次超过阈值
- `sum_best_ask` 首次跌破 `1 - buffer`
- `sum_best_bid` 首次高于 `1 + buffer`

每条记录至少包含：

- `market_id`
- `phase`
- `ts`
- `entry_type` (`YES / NO / FULLSET_BUY / FULLSET_SELL`)
- `entry_px`
- `theo`
- `sigma_short`
- `sum_best_ask / sum_best_bid`
- `binance features`
- `chainlink features`
- `H=10s/30s/60s/120s` 的 exit 和 pnl
- `fair_gain / convergence_gain / MFE / MAE`

### 实验 2：先把 phase 拆开，不要混跑

至少拆成：

- `PREWARM`
- `ACTIVE`
- `FAST_CLOSE`

这是必须的，不然你会把终局收敛和中段噪声混在一起。

### 实验 3：单独做 full-set 看板

这张表最可能最早出结果。

重点看：

- 事件频率
- 折价 / 溢价幅度
- 出现时的 depth 状态
- 出现 phase
- 是否集中在某几个 regime

### 实验 4：ACTIVE 段只做 realization filter

当前不要再问：

- “edge 大不大”

而要问：

- “这个 edge 未来 30s / 60s 会不会兑现”

label 可以直接定义成：

```text
future_convergence_60s > x
且 MAE_60s < y
```

### 实验 5：maker quote 只做质量分析，不做收益幻觉

可以先回答：

- 哪类 quote reason 后续最容易 adverse
- quote 离 top 太近是不是更危险
- `sigma_short` 高时是否更容易被打脸

但不要在没有 fills 的前提下把它包装成真实收益。

## 10. 现在最重要的认知升级

当前正确的升级方向不是：

```text
fair → edge → trade
```

而是：

```text
数据资产盘点
→ 按 phase 分层
→ 定义模拟可成交回放
→ 把收益拆成 fair drift / convergence / full-set
→ 只保留历史上真能兑现的结构
→ 再决定是否交易
```

一句话总结：

> 你现在最需要的不是”更大的 edge”，而是”哪些 edge 在当前 phase、当前流动性、当前外部 regime 下，真的会在可交易时间窗里兑现”。

如果只看当前数据，我建议优先顺序就是：

1. `full-set` 错价
2. `FAST_CLOSE` 特殊收敛
3. `ACTIVE` 的 realization filter
4. maker quote 质量
5. 暂时不要继续做 `PREWARM` raw edge taker

## 11. B 类机会验证（实证结论：B 类作为独立策略不存在）

### 11.1 B 类的定义与假设

B 类机会的原始假设是：

- 市场报价偏离了 theo，不是因为 theo 变了，而是因为盘口暂时没人报（流动性缺口）
- ask 临时下跌，bid 基本稳住（不对称 dislocation）
- 做市商回来后 ask 恢复，bid 小幅上移，roundtrip 可以盈利

为了让 B 类真正可交易，需要满足：

```
required_bid_recovery = entry_ask - current_bid + cost_buffer
```

只有 bid 实际回升幅度 > required_bid_recovery，这笔交易才有意义。

B 类进一步拆分为：

- **B1**：ask 下跌但幅度不够，roundtrip 成本覆盖不了，实际不可交易
- **B2**：ask 下跌幅度足够大，bid 基本没动（不对称），且靠近 full-set 边界

### 11.2 验证方法

用 1,089 个市场的 ACTIVE 阶段 book_top 数据，扫描所有 `Δask_15s < -0.02` 的事件，计算：

- `Δbid_15s`：bid 在同期的变化量
- `asym_ratio = Δbid_15s / Δask_15s`：接近 1 = 对称，接近 0 = 不对称（B2）
- 30s / 60s 后的 `bid_recovery`
- 对应的 `ask_sum`

### 11.3 实证结果

**核心数字：**

```
ask 下跌事件总数:     769,952
其中 bid 同步下跌：   766,328  （99.5%）  ← 对称
其中真正不对称：        3,624  （0.5%）   ← B2 候选
```

**各跌幅档位的 bid 跟跌情况（中位 asym_ratio 全部 = 1.000）：**

| ask 跌幅 | N | bid 跌幅中位 | asym_ratio 中位 | ask_sum 中位 |
|---|---|---|---|---|
| [0.02, 0.03) | 115,478 | -0.0200 | 1.000 | 1.0100 |
| [0.03, 0.05) | 217,109 | -0.0400 | 1.000 | 1.0100 |
| [0.05, 0.08) | 183,881 | -0.0600 | 1.000 | 1.0100 |
| ≥ 0.08 | 253,484 | -0.1200 | 1.000 | 1.0100 |

**B2 候选（3,624 个）的 bid 回升情况：**

| ask_sum 区间 | N | required_recovery | 实际 30s bid 回升 | 30s 盈利率 |
|---|---|---|---|---|
| [0.95, 0.99) | 0 | — | — | — |
| [0.99, 1.01) | 110 | 0.0045 | +0.0749 | **53%** |
| [1.01, 1.05) | 3,383 | 0.0104 | -0.0050 | 38% |
| ≥ 1.05 | 1 | 0.0100 | -0.0800 | 0% |

**theo 稳定性的影响（在 B2 候选内）：**

| theo 状态 | N | 30s 盈利率 | avg bid 回升 |
|---|---|---|---|
| 稳定（\|Δtheo\| < 0.005） | 494 | 43% | -0.0065 |
| 不稳（\|Δtheo\| ≥ 0.005） | 3,015 | 37% | -0.0024 |

### 11.4 结论

**B 类作为独立策略不存在。**

原因是 tick 结构：这个市场 spread 固定为 0.01，价格以 0.01 为单位整体移动，bid 和 ask 是同一个”报价中心”的两侧，不会分开。”ask 单独掉、bid 稳住”在这个市场结构里基本不发生（仅 0.5%）。

那 0.5% 的不对称事件里：
- 93% ask_sum > 1.01，30s 盈利率只有 38%，bid 均值继续跌 → 无价值
- 7%（110 个）ask_sum 在 [0.99, 1.01)，30s 盈利率 53%，bid 均值回升 +0.075 → **这是 C 类（full-set）的临界区，不是独立的 B 类**

**修正后的机会分类：**

- B 类不再单独建模
- 所有看起来像 B 类的点，实质上要么是整体价格移动（无 alpha），要么是 C 类的入口
- 策略研究直接聚焦 **C 类（full-set 错价）** 和 **FAST_CLOSE 终局收敛**

## 12. C 类验证（实证结论：full-set 套利在成熟市场不可执行）

### 12.1 验证数据

对 1,127 个市场（含 1,065 个已结算）扫描 ask_sum 和 bid_sum 分布（基于 50 个市场样本，329,198 个配对快照）：

**ask_sum（买 full-set 方向）：**

| 区间 | 数量 | 占比 |
|---|---|---|
| < 0.99 | 5 | 0.00% |
| < 1.00 | 44 | 0.01% |
| < 1.01 | 4,077 | 1.24% |
| 中位数 = 1.01 | — | — |

**bid_sum（卖 full-set 方向）：**

| 区间 | 数量 | 占比 |
|---|---|---|
| ≥ 1.00 | 619 | 0.19% |
| ≥ 1.01 | 73 | 0.02% |
| 最高值 = 1.07 | — | — |

### 12.2 结构性原因

Tick size = 0.01，决定了 ask_sum 的下限几乎固定在 0.99（即 0.49+0.50 的极端情形），正常状态下 ask_sum 中位数 = 1.01，**买 full-set 套利结构上不可能出现。**

bid_sum > 1.00 虽然有 619 个 tick（0.19%），理论上可以通过"铸造 YES+NO，再同时卖出"实现套利：

```
铸造 full-set：付 1.0 USDC → 得到 1 YES + 1 NO
卖出 YES（hit bid_yes）+ 卖出 NO（hit bid_no）
收回 bid_sum > 1.00，净利润 = bid_sum - 1.0
```

### 12.3 为什么不可执行

1. **铸造延迟**：mint YES+NO 需要链上操作，执行延迟足以让机会消失
2. **市场已成熟**：只要 bid_sum > 1.00 出现，专业套利者会立刻抹平，普通策略根本轮不到
3. **那 619 个 tick 是"正在被套利的瞬间"**，不是可进入的窗口

### 12.4 结论

> **C 类 full-set 套利在当前成熟市场环境下不可执行，无论买方向还是卖方向。不再作为独立策略建模。**

## 13. 策略地图（最终更新）

经过 B 类和 C 类完整验证，当前可研究的机会收窄为：

| 策略类型 | 状态 | 结论 |
|---|---|---|
| B 类：流动性缺口 | ❌ 已验证无效 | bid/ask 同步移动，不对称极少，无独立 alpha |
| C 类：full-set 套利 | ❌ 已验证无效 | 市场成熟，机会被即时清除，不可执行 |
| FAST_CLOSE 终局收敛 | ❌ 已验证无效 | poly 反应极快，BTC 方向明确时 ask=1.00 depth=0，无法进场 |
| **ACTIVE realization filter** | ⚠️ 需精细分层 | raw edge 整体为负，但特定条件下可能存在盈利点 |
| PREWARM raw edge | ❌ 不做 | 已有样本证明收益差，开盘噪声过多 |
| Maker quote 策略 | ⚠️ 独立分析 | 需要真实 fill 数据，当前只能做质量分析 |

**下一步优先级：**

1. **ACTIVE realization filter**：sigma 低 + theo 稳 + Binance 中性的组合条件下 edge 兑现率
2. **Maker quote 策略**：不依赖方向预测，靠 spread capture，核心风险是 adverse selection

## 13b. FAST_CLOSE 验证详情（实证结论：定价滞后不存在）

### 数据来源

1,065 个已结算市场，FAST_CLOSE 阶段全量快照，共 21,419 条。用 Binance 价格确定真实结果（UP/DOWN），记录赢边 ask 和等结算 P&L。

### 实证数据

**按距结算时间分桶：**

| 时间窗口 | N | correct_ask 均值 | pnl 均值 | % > 0 | depth 均值 |
|---|---|---|---|---|---|
| 0-15s | 9,560 | 0.8826 | 0.1174 | 81% | 443 USDC |
| 15-30s | 11,859 | 0.8431 | 0.1569 | 89% | 465 USDC |

**按 BTC 方向强弱分层（最后 60s）：**

| BTC 走势 | N | correct_ask | pnl | % > 0 | depth |
|---|---|---|---|---|---|
| BTC 涨 ≥ 0.3% | 257 | 1.0000 | 0.0000 | 0% | **0** |
| BTC 涨 0.1-0.3% | 1,842 | 0.9947 | 0.0053 | 59% | 724 |
| BTC 中性 ±0.1% | 17,568 | 0.8308 | 0.1692 | 94% | 436 |
| BTC 跌 0.1-0.3% | 1,415 | 0.9989 | 0.0011 | 45% | 536 |
| BTC 跌 ≥ 0.3% | 337 | 1.0000 | 0.0000 | 0% | **0** |

**BTC 方向明确时 poly 定价情况：**

| BTC 方向强度 | N | ask ≥ 0.97（已充分定价）|
|---|---|---|
| \|ret\| ≥ 0.1% | 3,851 | 100% |
| \|ret\| ≥ 0.3% | 594 | 100% |

### 核心悖论

- **BTC 方向明确**：poly 已完全定价（ask=1.00，depth=0），无法进场
- **BTC 方向中性**：94% 显示"盈利"，但这是**后验幻觉**——BTC ±0.1% 时 UP/DOWN 各报约 0.50，无论哪边赢买那边都便宜，但进场时不知道买哪边

### 结论

> Polymarket 对 BTC 价格变动反应极快。BTC 一旦方向明确，赢边 ask 立即到 1.00 且 depth 清零。FAST_CLOSE 收敛 alpha 不存在。

## 14. 新的判断：显性错价不是优势，优势要来自状态切换

经过 B 类和 C 类的排除，现在有一个更重要的认识：

> **市场上最显眼、最容易定义的错价，往往也是最早被做完的。**

这意味着后续研究重点不应该继续停留在：

- raw edge 多大
- ask_sum / bid_sum 有没有穿阈值
- 尾盘是不是普遍更好做

而应该转向：

- **哪些状态切换会短暂制造可交易的不连续**
- **哪些事件会让 Polymarket 的更新速度、库存状态、报单行为出现失真**
- **这种失真是否持续到足够让我们成交并退出**

换句话说，真正值得挖的不是“价格错了”，而是：

> **市场从一种状态切到另一种状态时，短时间内谁先动、谁后动、谁不动。**

## 15. 更深层的交易点：从静态价差转向事件驱动

如果市场已经比较成熟，那么可研究的 alpha 更像是“短窗口事件”，而不是“长期挂在那里的明显套利”。

我认为最值得研究的不是新 A/B/C 类，而是下面这些 **事件型机会**。

### 15.1 Phase Transition Event

不是看 `PREWARM / ACTIVE / FAST_CLOSE` 哪个 phase 平均更赚钱，而是看：

- `DISCOVERED → PREWARM`
- `PREWARM → ACTIVE`
- `ACTIVE → FAST_CLOSE`

这些切换前后几秒到几十秒，盘口和 theo 是如何重新对齐的。

研究重点：

- 切换前后 `book_top / depth / theo / quote_plan` 的跃迁
- 哪些 phase 切换后会出现短暂 stale quote
- 切换后 first refresh 的方向和速度

### 15.2 Quote Freeze / Refill Event

这类机会的本质不是“fair edge”，而是：

- Binance 已经动了
- Polymarket top/depth 暂时不动，或者动得很慢
- 然后某一时刻一次性补价或补深度

研究重点：

- Binance impulse 后，Polymarket 多久才刷新
- 刷新是一步到位，还是分多步追价
- refill 之后是 overshoot、undershoot，还是刚好回到合理区间

### 15.3 One-Sided Depletion Event

不是 ask 掉了没，而是：

- 某一侧 depth 被连续扫穿
- top 价变了，但补单速度跟不上
- 之后 1s / 3s / 10s 内会不会出现 inventory refill

研究重点：

- 扫穿前后的 depth slope
- refill 速度
- refill 后 bid/ask 的恢复路径

### 15.4 Adjacent Market Handoff Event

这是我认为非常值得挖的一类。

你现在做的是 BTC 15min 市场，天然存在：

- 当前合约即将结束
- 下一个合约刚被发现 / 刚进入 prewarm / 刚进入 active

这两个 market 之间理论上应该有很强的连续性，但在真实市场里不一定：

- 老 market 已经接近终局
- 新 market 刚开始建库存和建报价
- 做市商和普通交易者的注意力可能并不连续

研究重点：

- 相邻两个 market 在交接时刻的 theo、top、depth 是否平滑
- 新 market 刚激活后的前 10s / 30s 是否存在系统性定价偏移
- 老 market 的最后 30s 与新 market 的前 30s 是否有联动失真

### 15.5 Quote Plan vs Market Event

你现在手里还有 `pricing_quote_plan`，这很有价值。

它可以回答的不是“我们想报什么价”，而是：

- 系统计划怎么报
- 市场当时实际是什么状态
- 报价计划和真实盘口之间有没有固定偏差

研究重点：

- `quote_plan` 变化是否领先或滞后于 `market_book_top`
- 某些 `reason` 是否系统性对应更差的后续表现
- `maker_active / maker_fast_close / maker_prewarm` 的质量差异

## 16. 新的研究框架：建立事件库，而不是继续扩展价差分类

后续不建议继续新增 D/E/F 这种“按价格形态命名的策略类型”。

更好的方式是建立一个统一的 **事件库**，每种事件都用同一套框架验证。

建议先收集这些事件：

- `market_activate`
- `market_fast_close_enter`
- `binance_impulse_without_poly_refresh`
- `poly_refresh_after_lag`
- `one_sided_depth_depletion`
- `depth_refill_after_gap`
- `adjacent_market_handoff`
- `quote_plan_jump`

每个事件统一回答三件事：

1. 未来 `1s / 3s / 10s / 30s` 会发生什么
2. 哪个方向最稳定
3. 可成交 size 是否足够

也就是从：

```text
这个价格看起来便宜吗？
```

切换到：

```text
这个事件发生后，市场会如何重定价？
```

## 17. 事件研究的最小落地方式

为了避免继续停留在概念层，后续每个事件都只做一个最小闭环：

### 17.1 先定义事件触发

例如：

- Binance 5s return 超过阈值，但 Polymarket 3s 内 top 没更新
- `ACTIVE → FAST_CLOSE` 切换后的前 10s
- 某一侧 top depth 被连续打掉但 3s 内没回补

### 17.2 再做事件后路径统计

统一统计：

- mid / bid / ask 后续路径
- depth 恢复路径
- theo 是否继续同方向
- event 后的 MFE / MAE

### 17.3 最后只留下“事件后路径稳定”的结构

如果一个事件发生后：

- 方向经常反复
- 可成交 size 很差
- 持续时间太短

那就直接排除。

## 18. 最终更新后的主线

当前不应该再把主要精力放在：

- B 类流动性缺口
- C 类 full-set 套利
- 普通意义上的“raw edge 变大了”

而应该把主线改成：

1. **FAST_CLOSE 终局收敛**
2. **ACTIVE 中的事件驱动 realization filter**
3. **相邻 market 交接事件**
4. **Binance 先动、Polymarket 后补的更新滞后事件**
5. **单边 depth 枯竭与 refill 事件**

一句话总结：

> 如果显性错价已经被做干净，后续优势就不会来自“谁看到了更大的 edge”，而会来自“谁更早识别到市场状态刚刚发生了变化”。

|---|---|

- `1s`

| 字段 | 含义 |

这样后续筛事件时，不用再每次重写一套逻辑。

## 20. 事件类型的最小定义

下面这些事件，建议先做成 v1。

### 20.1 `market_fast_close_enter`

触发：

- `phase` 从 `ACTIVE` 切到 `FAST_CLOSE`

目的：

- 看进入终局后，book / theo / quote plan 的 first response

### 20.2 `binance_impulse_without_poly_refresh`

触发：

- Binance `5s return` 超过阈值
- 同时 Polymarket top 在接下来 `x ms` 内没更新或只更新一侧

目的：

- 找“外部先动，内部后补”的滞后窗口

### 20.3 `poly_refresh_after_lag`

触发：

- 发生过 `binance_impulse_without_poly_refresh`
- 然后 Polymarket 首次明显刷新

目的：

- 看是一步补齐、分步追价，还是补过头

### 20.4 `one_sided_depth_depletion`

触发：

- 某一侧前 1-2 档深度在短时间内被明显打掉
- 但另一侧没同步变化

目的：

- 看这是信息驱动还是库存冲击

### 20.5 `depth_refill_after_gap`

触发：

- `one_sided_depth_depletion` 之后
- 某一侧 depth 在短窗内明显恢复

目的：

- 看 refill 是否带来价格回补

### 20.6 `adjacent_market_handoff`

触发：

- 当前 market 距结束不足阈值
- 相邻下一个 market 已 discovered / prewarm / active

目的：

- 看相邻 market 交接时，定价和库存是否断裂

### 20.7 `quote_plan_jump`

触发：

- `pricing_quote_plan` 在短窗内出现显著跳变
- 但 market top 未同步变化，或变化方向不同

目的：

- 看系统计划与真实市场之间有没有固定错位

## 21. 扫描脚本设计（v1）

建议直接新增一个脚本：

```text
scripts/scan_event_opportunities.py
```

### 21.1 设计目标

这个脚本不负责下结论，只负责：

1. 扫描事件
2. 输出统一事件表
3. 生成每类事件的基础统计摘要

### 21.2 复用现有代码

当前已经有两个很适合复用的脚本：

- `scripts/build_dashboard_data.py`
- `scripts/compare_theo_models.py`

v1 最好直接复用其中的：

- `_discover_record_files`
- `_load_market_metadata`
- `_load_pair_book_series`
- `_load_theo_series`
- `_load_quote_series`
- `_load_latest_depth`

这样可以避免重新造一套 records 解析器。

### 21.3 命令行参数

建议的 CLI：

```bash
python scripts/scan_event_opportunities.py \
  --records-root runtime_data/records \
  --event-type market_fast_close_enter \
  --output runtime_data/research/events/market_fast_close_enter.jsonl
```

额外参数建议支持：

- `--event-type all`
- `--market-id <id>`
- `--limit-markets <n>`
- `--horizons 1000,3000,10000,30000`
- `--output-summary <path>`

### 21.4 输出格式

建议输出两类文件。

第一类：事件明细

```text
runtime_data/research/events/<event_type>.jsonl
```

每行一条事件记录。

第二类：汇总结果

```text
runtime_data/research/events/<event_type>.summary.json
```

包含：

- 事件数
- 去重后的独立事件数
- 不同 phase 分布
- horizon 后平均路径
- MFE / MAE 摘要
- 可成交 size 摘要

### 21.5 脚本处理流程

v1 可以按下面这条路径做：

```text
discover files
→ load global series
→ iterate markets
→ build aligned market timeline
→ detect events
→ attach forward-path labels
→ emit jsonl
→ aggregate summary
```

### 21.6 时间对齐原则

这是脚本里最容易出错的地方，必须先定规则：

- 事件时间以 market 数据的 `ts_ms` 为主
- 外部 Binance / Chainlink 一律取 `<= trigger_ts_ms` 的最近值
- forward label 用 `<= trigger_ts_ms + horizon` 的最近一条 market 记录
- 没有未来数据就标 `null`，不要强行插值

### 21.7 去重原则

事件不能每个 tick 都记一条。

建议：

- 同一 `market_id + event_type` 下设置 cooldown
- 或者用 “首次触发 + 恢复后再次触发” 机制

例如：

- `binance_impulse_without_poly_refresh`：cooldown `3s`
- `one_sided_depth_depletion`：cooldown `5s`
- `adjacent_market_handoff`：每对相邻 market 只保留一条主事件

### 21.8 v1 的实现优先级

不要一口气全写。

建议顺序：

1. `market_fast_close_enter`
2. `adjacent_market_handoff`
3. `binance_impulse_without_poly_refresh`

原因：

- 这三类最贴近当前主线
- 现有数据已经足够
- 比 depth 级别事件更容易先做出稳定结果

## 22. 真正的下一步，不是再想策略名，而是把事件扫出来

到这里为止，下一步已经很明确了：

- 不再继续扩展静态价差分类
- 不再纠结 B/C 是否还能挽救
- 直接开始构建 **事件库**

后续真正有可能形成优势的，不是：

- “发现了一个别人也知道的价差”

而是：

- “把市场状态切换事件系统化，知道哪些事件后面真的会出现稳定重定价路径”

所以接下来最该做的，是先把：

- `market_fast_close_enter`
- `adjacent_market_handoff`
- `binance_impulse_without_poly_refresh`

这 3 个事件扫出来，形成第一版 event library。
