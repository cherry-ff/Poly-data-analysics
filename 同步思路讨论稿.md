# 同步思路讨论稿

## 1. 讨论目标

这份文档只讨论一件事:

- 服务器已经写到磁盘上的数据，应该如何稳定地同步到本地，并在确认本地落盘后安全删除服务器上的已归档数据

这里要严格区分两条链路:

1. 采集链路  
   决定服务器当前会去订阅哪些 market，并持续把事件写入磁盘
2. 同步链路  
   决定本地如何读取服务器已经存储好的 segment，并在确认后做远端 GC

当前我们的核心判断是:

- 同步链路不应该依赖 discovery 当前发现了多少个 market
- 同步链路应该只依赖“服务器已经存储出来的 segment”


## 2. 当前正确的同步边界

### 2.1 sync 读取的对象

当前 sync API 读取的是:

- `runtime_data/records/sealed`

而不是:

- `runtime_data/records/live`
- 当前 registry 里有哪些 market
- 当前 discovery 正在跟踪哪些 market

这条边界是正确的，因为:

- `live` 文件仍在持续写入，不能直接同步后删除
- `sealed` segment 是稳定的、不可变的，才适合同步和删除


### 2.2 当前数据结构

服务端 recorder 的逻辑应该是:

1. 先写 live 文件
2. 满足轮转条件后切成 sealed segment
3. sync API 只导 sealed segment

因此，本地看到的数据总是会比服务器正在实时写入的数据“慢半步”。  
这不是 bug，而是为了保证同步和删除的边界清晰。


## 3. 当前同步链路

### 3.1 服务端

服务端当前同步链路大致是:

1. `manifest`
   列出 `sealed` 目录下所有可同步的 `.jsonl`
2. `archive`
   把请求的 segment 打成 `tar.gz`
3. `ack`
   客户端本地校验成功后，回传这批 segment 的 `path + sha256 + size_bytes`
4. `gc`
   服务端在 grace period 之后删除已经 ack 的 sealed segment


### 3.2 本地客户端

本地客户端当前同步链路是:

1. 拉 `manifest`
2. 从 manifest 里找出“本地还没确认过”的 segment
3. 按批次请求 archive
4. 下载到本地临时文件
5. 解压到 staging 目录
6. 校验 `size_bytes + sha256`
7. 原子移动到正式输出目录
8. 写本地 receipt 和 index
9. 如果带 `--delete-remote`，再提交 ack
10. 之后由服务端 GC 删除

这条链路的核心是:

- 本地先确认写成功
- 服务端再删除
- 删除发生在 sealed segment 层，不发生在 live 文件层


## 4. 当前已经确认正确的点

### 4.1 不删除 live 文件

这是必须坚持的原则。

如果对仍在写入的 live 文件做删除、截断、覆盖，会直接破坏数据连续性。


### 4.2 sync 不应依赖 discovery 数量

`POLY15_PM_DISCOVERY_MAX_MARKETS` 控制的是:

- 采集端当前最多会主动发现并注册多少个 market

它不应该决定:

- sync API 能同步多少个 market

sync API 应该只看磁盘上已经存在的 sealed segment。


### 4.3 大同步必须分批

全量 archive 太大时，容易在服务端打包阶段或 Nginx 反代阶段出现:

- timeout
- 502 Bad Gateway

因此同步客户端当前应该按批次拉取，而不是一口气请求一个超大 archive。

当前思路是:

- 按 `batch_max_bytes`
- 按 `batch_max_files`

做自动分批。


## 5. 当前已经暴露出来的问题

### 5.1 采集侧和同步侧曾经被混淆

之前把 `POLY15_PM_DISCOVERY_MAX_MARKETS` 改成 `0` 的方向是错误的。

原因:

- 它会放开采集侧当前注册的 market 数量
- 可能把很多离开始时间很远的 future market 也提前注册进来
- 这会扩大服务器负担
- 但并不会真正解决 sync 读取已存储 segment 的问题

这条已经回退。


### 5.2 当前 discovery 还是只按“数量”控制，不按“时间窗口”控制

目前 discovery 更像:

- 最多注册前 N 个当前/即将开始的 market

但更合理的逻辑应该是:

- 只注册已经开始的 market
- 或距离开始时间足够近的 market
- 再在这个集合里做数量限制

也就是说，未来很久才开始的 market 不应该过早注册。


### 5.3 sync 仍然只同步 sealed，不同步最新 live 尾巴

这是当前设计的结果。

所以如果用户看到:

- 某个 market 最新一小段数据在网页里有
- 但本地刚同步完还没有

通常不是同步丢数据，而是:

- 这段数据还在 live
- 还没有被轮转成 sealed


## 6. 当前建议的总体方向

### 6.1 采集侧

采集侧应该解决的是“不要过早注册太多 future market”。

建议不要只依赖 `discovery_max_markets`，而是增加时间窗口参数，例如:

- `discovery_future_start_window_minutes`

含义类似:

- 已经开始的 market: 可以注册
- 距离开始时间小于某个窗口的 market: 可以注册
- 距离开始时间很远的 market: 不注册

然后在这个过滤结果上，再应用 `discovery_max_markets`


### 6.2 同步侧

同步侧应该坚持:

- 只看已存储的 sealed segment
- 与 discovery 数量解耦
- 自动分批下载
- 本地校验后 ack
- 服务端 grace period 后 GC

同步侧不应该去推断:

- 当前系统理论上应该有多少 market
- Polymarket 页面上总共展示多少 market

同步侧只应该问一个问题:

- 服务器磁盘里当前已经 seal 出来了哪些 segment


## 7. 当前最推荐的讨论问题

接下来建议围绕这几个问题讨论:

1. 采集侧 future market 的“可注册时间窗口”到底应该设多大  
   例如 15 分钟、30 分钟、60 分钟，还是和市场周期绑定

2. 采集侧数量限制是否仍然保留  
   我倾向于保留，但放在“时间窗口过滤之后”

3. 服务端是否需要提供一个“强制 rotate 当前 live -> sealed”的接口  
   这样本地同步前可以先 seal 一次，减少“最新尾巴不同步”的感知

4. sync API 是否要增加“按时间范围/按 market 分组”的 manifest 视图  
   这样本地可以更明确地按旧数据优先同步

5. Nginx / dashboard 是否需要额外调整  
   例如更长的反代超时，或者服务端 archive 改为更细粒度流式生成


## 8. 当前结论

当前最重要的结论只有两条:

1. 同步应该基于”已存储的 sealed segment”，不是基于当前 discovery 数量
2. 采集侧需要一个更合理的”未来市场注册窗口”，而不是简单放开 `max_markets`

这也是接下来讨论和修正的主方向。

## 9. 基于代码的现状核查（实际实现 vs. 讨论假设）

> 本节是在完整读完代码之后补写的，目的是校正之前讨论中与实际代码不一致的地方。

### 9.0 实际目录结构（与讨论中的假设不同）

讨论中描述的是 `live/` 和 `sealed/` 两个子目录，但实际不是这样。

**实际的 recorder 写入路径（`storage/recorder.py`）：**

```
records/
├── global/
│   └── <topic>.jsonl           ← 当前正在写入的 live 文件（单个文件，无序号）
├── markets/
│   └── <market_id>/
│       └── <topic>.jsonl       ← 同上，market 级别的 live 文件
└── sealed/
    ├── global/
    │   └── <topic>/
    │       └── 00000000000000000001.jsonl   ← rotate 后的 sealed segment（带序号）
    └── markets/
        └── <market_id>/
            └── <topic>/
                └── 00000000000000000001.jsonl
```

关键点：
- live 文件就是 `records/global/<topic>.jsonl`（路径里没有 `live/` 层）
- rotate 时：把这个文件原子 rename 到 `records/sealed/global/<topic>/<N>.jsonl`
- rotate 条件：超过 `rotate_interval_ms`（默认 5 分钟）或超过 `rotate_max_file_size_bytes`（默认 16MB）
- rename 是 OS 级别的原子操作，不会出现"中途状态"

**sync API 的 `_sync_root()` 返回的是：**

```python
self.records_root / "sealed"
```

因此 manifest 里的 relative path 是相对于 `sealed/` 的，例如：
```
global/feeds_binance_tick/00000000000000000001.jsonl
markets/1626443/feeds_polymarket_market_book_top/00000000000000000001.jsonl
```

**本地同步后的目录结构：**

```
<output_dir>/                          ← 默认 local_sync_client/synced_records/
├── global/
│   └── feeds_binance_tick/
│       └── 00000000000000000001.jsonl
├── markets/
│   └── <market_id>/
│       └── ...
└── .poly15_sync_state/                ← 同步状态目录
    ├── index.json                     ← 已确认同步的 segment 索引
    └── receipts/
        └── 20260318T114402Z.json      ← 每次同步的 receipt
```

注意：本地**没有** `sealed/` 层，直接是 `global/` 和 `markets/` 层。

---

### 9.1 现有同步流程的实际实现

**服务端（`dashboard/server.py`）：**

1. `sync_manifest()` → 遍历 `records/sealed/` 下所有 `.jsonl`，每个文件计算 `sha256 + size_bytes + mtime_ns`，返回 manifest
2. `create_sync_archive(paths)` → 把指定文件打成 `tar.gz` 临时文件，发给客户端；发完后删除临时文件
3. `ack_sync_entries(entries)` → 把 entries 写入 `gc_index.json`，设置 `delete_after = now + grace_seconds`（默认 6 小时）
4. `collect_pending_sync_garbage()` → 每 5 分钟自动跑一次，删除超过 grace period 的 acked 文件；如果文件校验不过则标记为 `stale` 跳过

**本地客户端（`local_sync_client/poly15_sync_client.py`）：**

1. 拉 manifest → 与本地 sync index 对比 → 得到 pending 列表
2. 按 `batch_max_bytes`（128MB）和 `batch_max_files`（64）分批
3. 每个 batch：
   - 请求 `archive(batch_paths)` → 下载到系统临时文件（`tempfile.NamedTemporaryFile`）
   - `_extract_archive()` → 解压到 output_dir 内的临时目录（`tempfile.mkdtemp(dir=output_dir)`）
   - `_build_verified_entries()` → 校验 sha256 + size
   - `_commit_verified_entries()` → `staged_path.replace(destination_path)`（原子 rename）
   - **staging 临时目录立即删除**（`shutil.rmtree(staging_dir, ignore_errors=True)`）
   - 写 receipt + 更新 sync index
   - 如果 `--delete-remote`：发送 ack

### 9.2 与之前讨论的差异点

| 话题 | 之前讨论的假设 | 实际代码的情况 |
|---|---|---|
| staging 是否持久化 | 认为 staging 下会有 `.tmp` 文件可以恢复 | staging 是 `tempfile.mkdtemp`，每次 batch 完成后立即 `rmtree` 删除，不持久化 |
| 中断恢复粒度 | 认为可以按单个 segment 恢复 | 实际是按 batch 恢复，batch 内某个文件失败则整个 batch 不写 index，下次重新下载 |
| GC 触发方式 | 认为需要客户端手动调用 gc | 实际有后台线程每 5 分钟自动 GC，grace period 是 6 小时 |
| live 文件路径 | 描述为 `live/<topic>.jsonl` | 实际是 `global/<topic>.jsonl`（无 `live/` 层） |
| sealed 路径 | 描述为 `sealed/<topic>.jsonl` | 实际是 `sealed/global/<topic>/<seq>.jsonl`（多了 topic 目录层和序号） |
| 本地存储路径 | 认为本地也有 `sealed/` 层 | 实际本地路径直接是 `global/<topic>/<seq>.jsonl`，没有 `sealed/` 层 |
| sha256 计算时机 | 认为在 manifest 生成时计算 | 确认：manifest 生成时对每个文件实时计算 sha256（注意：大量文件时这有性能代价）|

### 9.3 当前中断恢复的实际行为

当 sync 进程被中断（SIGKILL 或网络断开）时，会有哪些遗留状态？

| 中断时刻 | 遗留物 | 下次 sync 的处理 |
|---|---|---|
| 下载 archive 途中 | 系统 temp 目录里的 `.tar.gz` 临时文件（`/tmp/poly15-sync-xxx.tar.gz`） | `_request_to_file()` 的 except 块会 `unlink(missing_ok=True)` → 但 SIGKILL 时来不及清理 → 下次运行不感知这个残留，会重新下载 |
| 解压到 staging 途中 | `output_dir/poly15-sync-xxx/` 临时目录（部分解压的文件） | `_extract_archive()` 的 except 块会 `rmtree(staging_dir)` → 但 SIGKILL 时来不及清理 → 下次运行**不会**自动清理这个目录，会重新下载一个新 batch（老的 staging 目录会残留） |
| 校验 / commit 途中 | staging 目录里有完整文件，output_dir 里可能有部分 commit 的文件 | 本次 batch 对应的 index 条目未写入 → 下次重新下载整个 batch → 对于已 commit 的文件，`staged_path.replace(destination_path)` 会覆盖 → 正常 |
| receipt / index 写入途中 | 文件已 commit，但 index 未更新 | 下次认为这些 segment 未同步 → 重新下载 → 覆盖 → 正常（幂等） |

**当前代码的潜在问题：**

`output_dir` 下可能积累 `poly15-sync-xxx/` 孤儿 staging 目录，需要手动清理或启动时扫描清理。
这不影响数据正确性，但会浪费磁盘空间。

---

### 9.4 manifest sha256 计算的性能隐患

`sync_manifest()` 在每次请求时都对 sealed 目录下所有 `.jsonl` 文件**实时计算 sha256**（见 `_build_sync_manifest_entry()`）。

如果 sealed 目录下有几千个文件，每次拉 manifest 的开销会很大（磁盘 IO + CPU）。

**当前可接受的原因：**
- 目前 sealed 文件数量还不多
- manifest 请求频率低（每次 sync 才调一次）

**如果未来文件数量大了，应考虑：**
- 服务端维护一个 sha256 缓存（以 `path + mtime_ns + size_bytes` 作为 key），只对新文件计算
- 或者在 rotate 时就写入 sha256 sidecar 文件

## 10. 四个核心同步问题的详细讨论

### 10.1 问题一：多次同步对齐

**问题描述**

每次运行同步客户端时，服务端 manifest 可能包含数百甚至数千个 segment。
如何判断”哪些已经同步过了，哪些还没有”？

**设计方案：本地 receipt 是唯一真相源**

每个 segment 由 `path + sha256 + size_bytes` 三元组唯一确定。
同步成功后，把这三元组写入本地 receipt 文件（例如 `.sync_receipts/receipts.jsonl`）。

每次同步启动时:

```
pending = manifest 里的 segment - 本地 receipt 里已有的 segment
```

只处理 `pending` 部分，已有 receipt 的直接跳过。

**receipt 的写入时机必须严格：**

- 必须在文件从 staging 原子移动到正式目录之后才写 receipt
- 不能在下载过程中写 receipt
- 不能在校验之前写 receipt

这样保证：如果 receipt 存在，文件一定已经落盘且校验通过。

**对齐的判断标准：**

```
本地已对齐 = (本地 receipt 集合) ⊇ (服务端当前 manifest 集合)
```

注意：服务端在 ack + gc 之后会从 manifest 中移除该 segment，
因此”本地 receipt 里有，但服务端 manifest 里没有”是正常状态，不是错误。

---

### 10.2 问题二：同步中断后如何续传

**问题描述**

同步过程中网络中断、进程 kill、磁盘写满等情况，
下次重新运行同步时如何做到”不重复下载已完成的，只补齐未完成的”？

**各阶段的中断处理：**

| 中断发生时刻 | 遗留状态 | 下次同步的处理 |
|---|---|---|
| 下载未完成 | `staging/<seg>.tmp` 临时文件 | 检测到 `.tmp` → 删除 → 重新下载 |
| 下载完成未校验 | `staging/<seg>.jsonl` 未经校验 | 重新校验；校验失败则删除重下 |
| 校验通过未移动 | staging 里有完整文件 | 重新执行原子 move 到正式目录 |
| 移动完成未写 receipt | 正式目录里有文件，无 receipt | 重新计算 sha256 + size 对比 manifest，通过则补写 receipt |
| receipt 写完未 ack | receipt 存在，未 ack | 直接重发 ack，不需要重下 |

**实现上的关键：所有步骤都必须是幂等的**

- 重新下载同一个 segment → 覆盖 staging 临时文件 → 正常
- 重新校验 → 结果确定性相同 → 正常
- 重新写 receipt → 检查是否已存在相同记录 → 不重复写
- 重新发 ack → 服务端对已 ack 的 segment 应该返回 200/204 而不是报错

**staging 目录的作用：**

```
download → <staging>/<seg>.tmp
rename   → <staging>/<seg>.jsonl   (下载完成标志)
verify   → sha256/size 校验
move     → <output_dir>/<seg>.jsonl  (原子写入正式目录)
receipt  → 追加到 receipts.jsonl
```

staging 和 output_dir 必须在同一个文件系统上，
这样 rename (原子 move) 才能保证原子性。

---

### 10.3 问题三：实时采集过程中如何同步

**问题描述**

服务器上的 recorder 在持续写入 live 文件，
同时同步客户端在跑，会不会出现冲突？会不会丢数据？

**结论：live 和 sealed 天然隔离，不会冲突**

核心原因是 sealed 文件是不可变的：
- recorder 对 sealed 文件只做一次 rename（从 live rotate 过来），之后不再写
- sync 只读 sealed 文件，不碰 live
- 因此两者之间没有竞争写，不会有文件损坏

**唯一需要关注的竞争点：rotate 瞬间**

时序:

```
t0: recorder 将 live/A.jsonl rename 成 sealed/A.jsonl  ← rotate
t1: sync 拉 manifest，此时 manifest 已包含 sealed/A
t2: sync 请求 archive A → 服务端读 sealed/A → 正常

t0': sync 在 rotate 之前就拉了 manifest，manifest 里没有 A
     sync 跑完这一轮，A 不在本次 pending 里
     下次运行时 A 已经在 sealed，manifest 会包含它 → 下次会同步
```

结论：rotate 只会导致”这次 manifest 看不到某个刚 seal 的 segment”，
下次同步就会补上，不会永久丢失。

**服务端 archive 接口的健壮性要求：**

如果 sync 请求 archive 时，某个 segment 文件恰好正在 seal 过程中（极短窗口），
服务端应该:
- 要么等 seal 完成再返回
- 要么返回 404，让客户端重试

客户端对 404 的处理：
- 移除本次 pending 中的该 segment
- 下次运行时重新从 manifest 拉取

**sync 是否应该和采集完全并行运行？**

推荐：可以并行，无需加锁，因为操作的目录不同。
但建议 sync 的运行频率不要太高（例如不低于 5 分钟一次），
避免对服务端 archive 接口产生过多并发压力。

---

### 10.4 问题四：如何确保全量同步

**问题描述**

怎么知道”所有数据都已经同步到了本地”？
服务端 GC 删除后 manifest 里就没有了，还能追溯吗？

**问题分解**

“全量”实际上有两层含义：

1. **已 sealed 的全量**：manifest 里当前列出的 sealed segment，是否都同步了？
2. **历史全量**：过去某段时间内产生的数据，是否一条都没有漏掉？

对于第 1 层，可以直接比对 `manifest 集合 - receipt 集合`，差值即未同步的。

对于第 2 层，需要更严格的设计。

**服务端必须保证：在 ack 之前不从 manifest 中移除 segment**

正确的 GC 流程：

```
sealed/A.jsonl 存在
   → manifest 包含 A
   → sync 下载 A
   → 客户端校验成功
   → 客户端发 ack
   → 服务端进入 grace period（例如 1 小时）
   → grace period 结束后从磁盘删除 A
   → manifest 中移除 A（或标记为已 GC）
```

如果服务端提前从 manifest 中移除了 A（但磁盘上 A 还在），
客户端就再也不会感知到 A 的存在，造成静默漏同步。

**客户端如何自检”是否全量”：**

```python
pending = [s for s in manifest if s.path not in local_receipts]
if len(pending) == 0:
    print(“当前 manifest 里所有 segment 均已同步”)
```

这只能保证”当前 manifest 里的全量”，不能追溯 GC 之前的历史。

**如果要追溯历史全量，需要服务端提供 segment 历史记录**

方案：服务端维护一个 `segment_history.jsonl`，记录所有曾经 sealed 过的 segment（包括已 GC 的），格式:

```json
{“path”: “...”, “sha256”: “...”, “size_bytes”: 123, “sealed_at”: “...”, “gc_at”: “...”}
```

客户端用本地 receipt 集合对比 segment_history，就能知道有没有遗漏。

---

### 10.5 同步程序的整体实现建议

**状态机（每个 segment 的生命周期）**

```
PENDING
  │ 开始下载
  ▼
DOWNLOADING (.tmp 临时文件)
  │ 下载完成 rename
  ▼
STAGED (.jsonl 在 staging 目录)
  │ sha256 + size 校验通过
  ▼
VERIFIED
  │ 原子 move 到 output 目录
  ▼
COMMITTED
  │ 写入本地 receipt
  ▼
RECEIPTED
  │ 发送 ack（如果启用 --delete-remote）
  ▼
ACKED
```

重启后，通过扫描 staging 目录和 receipt 文件，可以恢复每个 segment 的当前状态，从中断点继续。

**同步主循环的伪代码**

```python
def sync_loop():
    manifest = fetch_manifest()           # 拉服务端 manifest
    receipts = load_local_receipts()      # 读本地 receipt
    pending = diff(manifest, receipts)    # 找出未同步的

    pending = sort_by_time_asc(pending)   # 优先同步旧数据

    for batch in chunk(pending, batch_max_bytes, batch_max_files):
        tmpfile = download_archive(batch)         # 下载到 .tmp
        segments = extract_to_staging(tmpfile)    # 解压到 staging
        for seg in segments:
            verify(seg)                           # sha256 + size
            atomic_move(seg, output_dir)          # 移到正式目录
            append_receipt(seg)                   # 写 receipt
        if delete_remote:
            ack(batch)                            # 回传 ack

def main():
    recover_interrupted_staging()  # 启动时清理/恢复 staging 状态
    sync_loop()
```

**并发与重试**

- 批次级别可以并发（多个 batch 同时下载），但建议并发数不超过 3
- 单个 batch 失败 → 最多重试 3 次，指数退避
- 整个 sync_loop 失败 → 进程级别的 retry 由外部调度（cron / supervisor）负责

**推荐目录结构**

```
<local_base>/
├── segments/          # 正式落盘目录（output_dir）
│   └── <market_id>/
│       └── <segment>.jsonl
├── staging/           # 临时下载和解压区
│   └── *.tmp / *.jsonl
└── .sync/
    ├── receipts.jsonl # 已确认同步的 segment 记录
    └── ack_queue.jsonl # 已 receipt 但未 ack 的（用于重启后补发 ack）
```

**关于 force_seal 接口的建议**

如果需要”尽量减少 live 尾巴未同步的感知”，
服务端可以提供一个 `POST /sync/force_seal` 接口：

- 把当前所有 market 的 live 文件手动 rotate 成 sealed
- 仅在手动触发全量同步前调用，不建议自动化调用
- 这个接口不影响数据完整性，只是提前 seal

这样在执行大批量同步之前先 force_seal 一次，
可以把”最新尾巴”也纳入本次同步范围。

| 优先级 | 问题 | 说明 |
|---|---|---|
| P0 | receipt 系统 + 幂等性 | 这是所有其他问题的基础，必须先做对 |
| P0 | 服务端 GC 必须在 ack 之后 | 否则可能永久丢数据 |
| P1 | staging 目录 + 中断恢复 | 保证续传可靠性 |

## 11. GB 级别数据量下的同步问题专项讨论

> 服务器每天产生几个 GB 数据，这是实际约束，需要把它当作设计前提来讨论。

### 11.1 先把数量级算清楚

**每天产生多少 sealed segment？**

录制参数（`app/config.py`）：
- `rotate_interval_ms = 300_000`（5 分钟）
- `rotate_max_file_size_bytes = 16 MB`

假设服务器同时跟踪 10 个 market，每个 market 有 5 个 topic
（book_top / depth / metadata / pricing_theo / pricing_quote_plan），
再加 global 3 个 topic（binance_tick / binance_depth / chainlink_tick）：

- 活跃 topic 数：10 × 5 + 3 = 53 个 topic
- 每个 topic 每天最多 rotate：24h × 60min / 5min = 288 次（按时间触发）
- 如果某个 topic 写入快，16MB 触发比 5 分钟更频繁

**粗估：每天 300～2000 个 sealed segment，原始总大小 3～5 GB**

这意味着：
- 一周不同步：2100～14000 个文件，21～35 GB
- 一个月不同步：9000～60000 个文件，90～150 GB

---

### 11.2 当前代码的核心瓶颈：manifest sha256 是 O(总数据量)

**代码路径（`dashboard/server.py:sync_manifest()`）：**

```python
for path in self._iter_syncable_files():       # 遍历所有 sealed 文件
    entry = self._build_sync_manifest_entry(path, sync_root)
    # 其中包含：
    # "sha256": self._file_sha256(path)         # 每次都重新读全文件计算
```

`_file_sha256()` 每次都完整读取文件内容。

**代价分析：**

| sealed 文件数 | 平均文件大小 | manifest 请求需要读的数据量 | 大致耗时（机械盘） |
|---|---|---|---|
| 300（1天） | 10 MB | 3 GB | ~30 秒 |
| 2100（1周） | 10 MB | 21 GB | ~3.5 分钟 |
| 9000（1个月） | 10 MB | 90 GB | ~15 分钟 |

**结论：manifest 接口在 GB 级别场景下是不可用的。**

每次同步客户端调 `/api/sync/manifest`，服务端都会把 sealed 目录里所有文件重新读一遍算 sha256。
而且这是在 HTTP 连接的 `do_GET()` 里同步执行的，会直接阻塞整个 HTTP 服务线程。

---

### 11.3 sealed 文件是不可变的，sha256 可以永久缓存

rotate 之后，sealed 文件永远不会被修改（只会被删除）。

这意味着：**`(path, size_bytes, mtime_ns)` 作为 key 的 sha256 缓存永远不会失效**。

一旦计算过某个 path 的 sha256，缓存就可以一直用到文件被 GC 删除。

**修复方案：服务端内存中维护 sha256 缓存**

```python
# 伪代码
class DashboardHTTPServer:
    def __init__(self, ...):
        self._sha256_cache: dict[str, tuple[int, int, str]] = {}
        # key: relative_path → value: (size_bytes, mtime_ns, sha256)

    def _file_sha256_cached(self, path: Path, sync_root: Path) -> str:
        stat = path.stat()
        relative = path.relative_to(sync_root).as_posix()
        cached = self._sha256_cache.get(relative)
        if cached and cached[0] == stat.st_size and cached[1] == stat.st_mtime_ns:
            return cached[2]
        digest = self._file_sha256(path)            # 只有第一次才读文件
        self._sha256_cache[relative] = (stat.st_size, stat.st_mtime_ns, digest)
        return digest
```

效果：
- 第一次 manifest 请求：读全部文件（同现在）
- 后续所有 manifest 请求：只读 stat（不读文件内容），O(文件数) 而非 O(总数据量)
- GC 删除文件后：下次 manifest 不再遍历到该 path，缓存自然失效

---

### 11.4 archive 是全量打包再发送，不支持流式

**代码路径（`create_sync_archive()`）：**

```python
with tarfile.open(archive_path, mode="w:gz") as archive:
    for path in files:
        archive.add(path, ...)
# 整个 tar.gz 写完之后，才开始 HTTP 响应
return archive_path, filename, archive_path.stat().st_size
```

这是先把整个 archive 写到磁盘，再发给客户端。

**问题：**
- 128 MB 原始数据 → gzip 压缩后约 30～80 MB archive
- 服务端要先写 30～80 MB 临时文件，再读它发出去
- 在发送期间，临时文件占用服务端磁盘
- 压缩耗时 + 写磁盘耗时 + 传输耗时 串行叠加

**在当前批次大小（128MB）下可接受，但有三个前提：**
1. 服务端磁盘有足够空间（至少 archive_size × 并发请求数）
2. Nginx 反代的 `proxy_read_timeout` 足够长（否则在服务端打包期间 Nginx 就超时了）
3. 客户端 `archive_timeout_seconds = 1800`（已经设得很长）

**建议：减小 batch_max_bytes 到 32～64 MB**

原因：
- 32 MB 原始 → 约 8～20 MB archive → 打包快、传输快、失败重试代价小
- 128 MB 批次失败需要重新下载 128 MB；32 MB 失败只需重新下载 32 MB
- 批次数量翻 4 倍，但每批风险更低，整体可靠性更高

---

### 11.5 manifest 里的排序问题

**当前 `_iter_syncable_files()` 的排序：**

```python
return sorted(path for path in sync_root.rglob("*.jsonl") if path.is_file())
```

这是按路径字母序排序，例如：
```
global/feeds_binance_depth/00000000000000000001.jsonl
global/feeds_binance_depth/00000000000000000002.jsonl
...
markets/1626443/feeds_polymarket_market_book_top/00000000000000000001.jsonl
...
```

**客户端当前的批次顺序 = manifest 顺序 = 路径字母序**

这意味着：
- binance 的数据总是在 polymarket 数据之前同步
- market_id 小的 market 总是在 market_id 大的 market 之前同步
- **不是按时间顺序同步**

**为什么顺序重要？**

manifest 中每个 entry 都有 `mtime_ns` 字段（文件 rotate 时间）。
如果按 `mtime_ns` 升序排列批次，就能实现"最老的数据优先同步"。

在 GB 级别的初始同步场景下，如果网络不稳定、同步被迫中断，
"最老优先"可以保证历史数据优先落盘，剩下没同步的是最新的数据
（而最新的数据还可能在 live 里，丢失风险相对低）。

**建议：客户端在 `_run_sync()` 中将 archive_paths 按 mtime_ns 升序排序：**

```python
archive_paths = sorted(
    archive_paths,
    key=lambda p: manifest_entries[p].get("mtime_ns", 0)
)
```

---

### 11.6 初始全量同步 vs. 增量同步的不同策略

**增量同步（正常运行期间，每隔几分钟跑一次）：**

- sealed 目录里只有少量新文件（上次同步以来产生的）
- manifest 返回的未同步 entry 数量少（几十个）
- 总数据量小（几百 MB 以内）
- 当前代码完全够用

**初始全量同步 / 长时间中断后恢复：**

- sealed 目录里有几十 GB 历史数据 → 数千个文件
- manifest sha256 计算极慢（见 11.3）
- 需要同步的 batch 数量多（几十到几百个）
- 每个 batch 的失败都会导致重试，累计时间很长

**初始全量同步的推荐做法：**

1. **先修复 manifest sha256 缓存**（11.3），否则每次 manifest 请求本身就要跑几分钟
2. **调小 batch_max_bytes**：初始同步时用 `--batch-max-bytes 33554432`（32MB）
3. **分阶段同步**：按 market_id 或按时间段手动分批
   ```bash
   # 先同步某个 market
   python poly15_sync_client.py sync markets/1626443
   # 再同步下一个
   python poly15_sync_client.py sync markets/1626576
   ```
4. **在低峰期运行**：避免和实时采集抢磁盘 I/O
5. **循环运行**：用 shell 脚本循环调用 sync，每次运行结束后等待一段时间再继续
   ```bash
   while true; do
     python poly15_sync_client.py sync --delete-remote
     sleep 60
   done
   ```

---

### 11.7 "已全部同步"的判断方法

当 sealed 目录里有很多文件时，怎么确认本地都同步了？

**方法：对比 manifest 中 `file_count` 与本地 index 中的条目数**

```bash
# 拉 manifest，看 file_count
python poly15_sync_client.py manifest | python3 -c "import json,sys; d=json.load(sys.stdin); print('remote:', d['file_count'], 'files,', d['total_bytes'], 'bytes')"

# 看本地 index
python3 -c "import json; d=json.load(open('.poly15_sync_state/index.json')); print('local:', len(d['entries']), 'confirmed')"
```

如果 `remote file_count == local confirmed count`（且 sha256 全部匹配），则全量同步完成。

注意：GC 删除已 acked 的文件后，`remote file_count` 会减少，但本地 index 里仍保留记录。
所以只能用"本地 index 是 manifest 的超集"来判断，不能用"数量相等"来判断。

---

### 11.8 当前最关键的 3 个改动

基于上述分析，按优先级排列：

**P0：服务端 manifest sha256 缓存**
- 不做这个，GB 级别数据时 manifest 请求会超时
- 改动范围：`dashboard/server.py`，在 `_build_sync_manifest_entry()` 里加缓存逻辑
- 不涉及数据格式变化，向下兼容

**P1：客户端按 mtime_ns 排序 archive_paths**
- 保证大量数据时"老数据优先同步"
- 改动范围：`local_sync_client/poly15_sync_client.py`，在 `_run_sync()` 里加一行 sort

**P1：调小默认 batch_max_bytes（或在运行脚本里显式指定）**
- 当前默认 128MB，建议改为 32MB 或在实际运行时传参
- 不改代码也可以，用 `--batch-max-bytes 33554432` 参数解决

## 12. 进一步分析与讨论结论（基于当前代码与最近实际运行）

> 本节不是再提出一套新的理想方案，而是对上面已有讨论做收敛，明确“哪些判断已经成立，哪些才是下一步最值得讨论的点”。

### 12.1 已经可以确认的结论

#### 结论一：同步链路和 discovery 链路必须彻底分开

这是当前最重要的结论。

- discovery 决定的是：服务器现在去订阅哪些 market
- sync 决定的是：服务器磁盘里已经 seal 出来的哪些 segment 要搬到本地

因此：

- `POLY15_PM_DISCOVERY_MAX_MARKETS` 不应该被当作同步问题的修复手段
- 即使服务器页面上“理论上有很多 market”，sync 也只应该看 `sealed` 目录里真实存在的 segment

换句话说：

**sync 不负责推断“应该有多少 market”，只负责同步“已经被 recorder 落盘的 segment”。**


#### 结论二：只同步 sealed 是正确边界

当前设计里：

- dashboard 页面读的是 `live + sealed`
- sync API 读的是 `sealed`

这会带来一个表象：

- 网页里能看到某个 market 的最新数据
- 但刚跑完 sync，本地还没有最后一小段

这通常不是 bug，而是因为：

- 那一小段还在 live
- 还没有 rotate 成 sealed

这条边界本身是合理的，不能为了“看起来同步得更全”就去直接导出 live 文件。


#### 结论三：当前真正先暴露出来的问题是“大归档不稳定”

从最近的实际运行看：

- `manifest` 已经能正常返回
- 真正失败的是 `archive`
- 失败形式包括：
  - 客户端等待服务端打包超时
  - 经 Nginx 反代后返回 `502 Bad Gateway`

这说明当前最直接的问题不是“发现几个 market”，而是：

- 单次 archive 太大
- 服务端打包耗时太久
- 代理链路对长时间无响应的大请求不稳定

也就是说，**同步侧当前最该先解决的是批次、顺序和 manifest 性能，而不是继续动 discovery 数量。**


### 12.2 当前文档里哪些部分值得保留，哪些要谨慎看

#### 值得保留的部分

下面这些判断已经和当前代码、当前运行现象对齐：

- sync 只应基于 sealed segment
- live 文件绝不能直接删除或裁剪
- 本地成功落盘后再 ack，服务端 grace period 后再 GC
- 大同步必须分批
- future market 不应过早注册


#### 需要谨慎看的部分

文档中有一些章节属于“理想化方案”，不是当前真实实现：

- 单 segment `.tmp` 状态机
- `receipts.jsonl`
- `ack_queue.jsonl`
- 启动时自动扫描并恢复 staging
- 按 segment 粒度断点续传

当前代码实际更接近：

- 以 batch 为单位下载 archive
- 以 batch 为单位解压到 staging 目录
- batch 成功后写 receipt/index
- batch 失败则下次重下整个 batch

这意味着后续讨论时要先明确：

- 我们是在讨论“当前已经实现的行为”
- 还是在讨论“下一版理想设计”

这两者不能混着用。


### 12.3 现在最值得继续讨论的三个问题

#### 问题一：同步顺序到底应该怎么排

当前服务端 manifest 顺序本质上是路径字典序，不是时间顺序。

这会带来一个问题：

- 初始大同步如果中途失败
- 先落到本地的不一定是最老的数据
- 可能只是某些路径字母序靠前的数据

从“服务器减压归档”的目标看，更合理的是：

- 优先同步最老的 sealed segment

可选方案有两个：

1. **全局最老优先**
   - 按 `mtime_ns` 升序同步所有 segment
   - 优点：最符合“先搬旧数据”的目标
   - 缺点：不同 market 会交叉进行，观察上不够直观

2. **按 market 成组，再组内最老优先**
   - 例如一次先把 `market A` 的历史补齐，再同步 `market B`
   - 优点：业务上更容易理解，也方便人工核对
   - 缺点：如果某个 market 特别大，会阻塞其他 market 的归档进度

这是当前最值得明确的一个策略问题。


#### 问题二：默认批次大小到底应该多大

从设计上讲，分批已经是正确方向。  
现在真正要讨论的是：

- 默认批次到底用 32MB、64MB，还是保留 128MB

我的倾向是：

- 初始全量同步：更适合 32MB 或 64MB
- 日常增量同步：128MB 也许可以接受

因为初始全量同步最怕的不是“批次数多”，而是：

- 任意一个大批次失败后，重试代价太高
- Nginx / upstream 在长时间打包阶段更容易返回 502

所以这个问题本质上是在平衡：

- 批次数量
- 单批失败代价
- 服务端打包稳定性


#### 问题三：采集侧 future market 的注册窗口应怎么定义

这里不应再围绕“无限发现”讨论，而应直接讨论一个更合理的注册窗口。

例如：

- 已经开始的 market：立即注册
- 距离开始时间小于 15 分钟：允许提前注册
- 距离开始时间很远：不注册

然后再叠加 `discovery_max_markets`

也就是：

1. 先做时间窗口过滤
2. 再做数量上限控制

这样比“只按数量截断”更符合业务含义，也能避免把大量 future market 过早挂进系统。


### 12.4 当前的优先级收敛建议

如果只按“最值得立刻推进”的顺序，我的建议是：

#### P0：让大同步稳定下来

优先处理：

- 批次大小
- archive 失败重试
- 代理链路 502 风险

这是用户已经实际撞到的问题，优先级最高。


#### P1：明确同步顺序

决定到底是：

- 全局最老优先
- 还是按 market 成组优先

这会直接影响“同步中断后，本地先拿到哪一部分历史”的行为。


#### P2：给 discovery 增加“未来开始时间窗口”

这一步是必要的，但它属于采集侧治理，不是当前同步失败的直接原因。

因此它应该做，但不应该排在大归档稳定性之前。


#### P3：manifest hash 缓存

从长期看，这一项非常重要。

只要 sealed 文件数持续增长，manifest 每次全量重算 sha256 就会越来越慢。

所以它虽然未必是当前最先爆炸的问题，但一定是中期必须处理的问题。


### 12.5 基于当前讨论的建议结论

把当前所有讨论压缩成一句话，就是：

**采集侧解决“哪些 market 现在值得注册”，同步侧解决“哪些已存储 segment 应该优先、稳定地搬走”，两者不能互相替代。**

因此下一轮讨论，建议只围绕这三个具体问题展开：

1. sync 默认按什么顺序拉取  
   是全局最老优先，还是按 market 成组优先
2. sync 默认 batch 大小定多少  
   是 32MB、64MB，还是保留 128MB
3. future market 的注册窗口定多少  
   是 15 分钟、30 分钟、60 分钟，还是与市场周期绑定

如果这三个问题先定下来，后续改造方向就会非常清晰。

### 13.2 502 的根本原因更可能是 Nginx 配置，不是代码

Codex 把 502 归因于"单次 archive 太大 / 服务端打包耗时太久"，这是对的。
但还有一个更直接、更容易修的原因没有说：

**Nginx `proxy_read_timeout` 默认只有 60 秒。**

当服务端正在打包 archive（读文件 + gzip 压缩 + 写临时文件），
如果这个过程超过 60 秒，Nginx 就会对客户端返回 `502 Bad Gateway`，
即使服务端本身没有崩溃、只是还在工作。

当前 `archive_timeout_seconds = 1800`（客户端等 30 分钟），
但如果 Nginx 只等 60 秒就断了，客户端的超时设置完全没有意义。

**查看当前 Nginx 配置：**

```nginx
# deploy/nginx/binance-monitor.conf 里看 /api/sync/archive 是否有单独配置
```

**最快的修法（不改代码）：**

在 Nginx 配置里为 sync archive 路由单独延长超时：

```nginx
location /api/sync/archive {
    proxy_pass         http://127.0.0.1:<port>;
    proxy_read_timeout 3600s;   # 给服务端足够时间打包
    proxy_send_timeout 3600s;
}
```

这比减小批次大小优先级更高，因为：
- 修 Nginx 配置不需要重启采集进程
- 减小批次大小只能降低单次打包时间，但无法保证不超过 60 秒

**结论：P0 应该是先查 Nginx timeout 配置，再讨论批次大小。**

### 13.3 manifest sha256 缓存的优先级应是 P1 而非 P3

Codex 说"manifest 已经能正常返回"，所以把 sha256 缓存放 P3。

这个判断依赖一个前提：**当前 sealed 目录里文件还不多**。

问题在于 sealed 文件数是单调递增的（直到 GC 后才减少），
而 GC 只在 ack 之后 6 小时才删文件。

**如果当前正在进行大批量初始同步，sealed 目录里可能已经有几百甚至上千个文件。**
每次 manifest 请求都要重新读所有这些文件计算 sha256，
这个 I/O 和当前正在进行的 archive 下载会产生磁盘 I/O 竞争。

更具体的问题：

```
时序：
  客户端调 manifest → 服务端读 500 个文件计算 sha256（可能 30 秒以上）
  客户端调 archive  → 服务端读同一批文件打包（又是几十秒）

两个操作同时发生时，磁盘 I/O 翻倍，archive 打包变慢，更容易超时。
```

所以 sha256 缓存不只是"文件多了以后才需要"的优化，
它还直接影响当前大同步期间的稳定性。

**建议把 sha256 缓存提升到 P1**，和"明确同步顺序"同级。

---

### 13.4 同步顺序：选全局最老优先，不选按 market 成组

Codex 给了两个方案，我的意见是：**选全局最老优先（按 mtime_ns 升序）**。

理由：

1. **GC 目标是释放服务器磁盘空间**。GC 触发条件是 ack，ack 触发条件是本地确认。
   因此只有已经同步 + ack 的文件才能被 GC。
   全局最老优先能让最早产生的数据最先离开服务器，磁盘释放速度最快。

2. **按 market 成组会导致某些 market 的旧数据长期卡在服务器**。
   如果 market A 数据很多，在它全部同步完之前，market B/C/D 的数据都不 ack，
   服务器磁盘空间没有实质性释放。

3. **mtime_ns 已经在 manifest 里**，客户端排序只需要一行代码，
   不需要服务端改任何东西。

唯一的反驳：全局最老优先会导致不同 market / topic 的文件交叉同步，
本地目录里可能出现"每个 market 都有一半数据"的状态。
但这是正常的，不影响数据可用性（每个 segment 独立可读）。

### 13.5 批次大小：推荐默认改为 64MB，初始同步用 32MB

Codex 的问题是"32 / 64 / 128 哪个合适"，我的判断：

**JSONL 文件 gzip 后压缩比约 5:1 到 10:1。**

- 128 MB 原始 → archive 约 15～25 MB → 网络传输快，但服务端打包需要 3～10 秒
- 64 MB 原始 → archive 约 8～12 MB → 服务端打包约 1～5 秒
- 32 MB 原始 → archive 约 4～6 MB → 服务端打包约 1～3 秒

**Nginx 默认 60 秒 timeout，服务端打包需要 3～10 秒（128MB），**
正常情况下不会超时。但如果服务器同时在采集（磁盘 I/O 竞争），打包可能变慢数倍。

建议：
- **默认 `batch_max_bytes` 改为 67108864（64MB）**
- 这覆盖大多数场景，既不会让批次数量爆炸，也减少了极端情况下的打包耗时
- 初始全量同步时，用 `--batch-max-bytes 33554432`（32MB）更保险

### 13.6 补充一个 Codex 没有讨论的问题：孤儿 staging 目录

Codex 没有提到这个，但它是实际运行中会出现的：

当 sync 进程被 SIGKILL 中断时，`output_dir` 下会遗留 `poly15-sync-xxx/` 临时目录。

**当前代码没有启动时清理逻辑。**

这些目录不影响数据正确性，但：
1. 会占用磁盘空间（最多 = 一个 batch 的解压后大小，即 32～128 MB）
2. 如果多次中断，会积累多个孤儿目录
3. 在磁盘本来就紧张的服务器减压场景下，这是个潜在问题

**解决方法（在 `_run_sync()` 开头加）：**

```python
# 清理 output_dir 下残留的 poly15-sync-* 临时目录
for stale in output_dir.glob("poly15-sync-*"):
    if stale.is_dir():
        shutil.rmtree(stale, ignore_errors=True)
```

这是一个小改动，但在初始大同步场景下很有必要。
