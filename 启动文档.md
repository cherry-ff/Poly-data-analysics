# Poly-15minBTC 启动文档

本文档用于说明这套系统在当前代码版本下的启动方式，以及数据同步、ack 和 GC 删除的标准操作流程。

## 1. 当前版本的数据流

当前版本的数据归档链路如下：

1. `collector` 持续写入 `runtime_data/records` 下的 live 文件。
2. `recorder` 按时间或文件大小把 live 文件轮转成 `sealed segments`。
3. `dashboard` 只对 `sealed` 数据暴露同步接口。
4. 本地同步客户端下载并校验归档数据。
5. 客户端把已校验的 segment 提交为 `ack`。
6. 服务端在 grace period 到期后，由 GC 删除这些已经 ack 的 `sealed segments`。

注意：

- 当前版本不会直接删除 live 文件。
- `sync --delete-remote` 不是立即删除，而是“提交 ack 并等待服务端 GC”。

## 2. 关键目录

- 项目根目录：`Poly-15minBTC`
- 运行时 records：`runtime_data/records`
- 服务端 sync token：`runtime_data/sync_api_token.txt`
- 服务端 ack / GC 状态：`runtime_data/sync_state/gc_index.json`
- 本地同步输出目录默认值：`local_sync_client/synced_records`
- 本地同步 receipt / index 默认目录：`<output-dir>/.poly15_sync_state`

## 3. 启动前准备

### 3.1 Python 环境

项目脚本默认使用：

- `PROJECT_ROOT/.venv/bin/python`

如果还没有虚拟环境，可以在项目根目录执行：

```bash
python -m venv .venv
.venv/bin/python -m pip install -e .
.venv/bin/python -m pip install pytest pytest-asyncio
```

### 3.2 环境变量

复制一份环境文件：

```bash
cp .env.example .env
```

最重要的几个配置：

- `POLY15_RECORDER_ENABLED=1`
- `POLY15_RECORDER_OUTPUT_DIR=runtime_data/records`
- `POLY15_RECORDER_ROTATE_INTERVAL_MS=300000`
- `POLY15_RECORDER_ROTATE_MAX_FILE_SIZE_BYTES=16777216`
- `POLY15_SYNC_API_TOKEN=`  
  为空也可以，`dashboard` 启动时会自动生成 token 并写入 `runtime_data/sync_api_token.txt`
- `POLY15_SYNC_DELETE_GRACE_SECONDS`  
  如果没有设置，`run_dashboard.sh` 默认用 `21600` 秒，也就是 6 小时
- `POLY15_SYNC_GC_INTERVAL_SECONDS`  
  如果没有设置，`run_dashboard.sh` 默认用 `300` 秒，也就是 5 分钟
- `POLY15_PM_DISCOVERY_MAX_MARKETS`  
  这个值只影响服务端采集端当前会主动发现并订阅多少个 market，不影响 sync API 遍历已经落盘的历史 segment。建议保持有限值，例如 `3` 或 `4`，避免把距离开始时间很远的 future market 过早注册进来

## 4. 服务端启动

### 4.1 启动 collector

```bash
bash scripts/run_collector.sh
```

作用：

- 启动主采集进程
- 持续写入 `runtime_data/records`
- 自动把 live 文件轮转成 `sealed segments`

### 4.2 启动 dashboard

```bash
bash scripts/run_dashboard.sh
```

作用：

- 启动本地 HTTP 服务
- 对外提供 dashboard 和 sync API
- 自动维护 `runtime_data/sync_state/gc_index.json`
- 后台定时执行 GC

默认监听：

- `127.0.0.1:18080`

如果需要修改端口：

```bash
export POLY15_DASHBOARD_PORT=18081
bash scripts/run_dashboard.sh
```

### 4.3 启动后检查

启动 `dashboard` 后，终端会输出：

- `records root`
- `sync token file`
- `sync state dir`
- `sync delete grace seconds`
- `sync gc interval seconds`

需要重点确认这几个路径是否正确。

## 5. 同步客户端使用

同步客户端脚本：

- `local_sync_client/poly15_sync_client.py`

可以直接用环境变量，也可以显式传 `--base-url` 和 `--token`。

### 5.1 查看 manifest

```bash
python local_sync_client/poly15_sync_client.py \
  --base-url http://127.0.0.1:18080 \
  --token "$(cat runtime_data/sync_api_token.txt)" \
  manifest
```

这个接口只会列出 `sealed` 文件，不会列出 live 文件。

### 5.2 同步全部数据

```bash
python local_sync_client/poly15_sync_client.py \
  --base-url http://127.0.0.1:18080 \
  --token "$(cat runtime_data/sync_api_token.txt)" \
  sync \
  --output-dir ./synced_records
```

### 5.3 只同步单个 market

```bash
python local_sync_client/poly15_sync_client.py \
  --base-url http://127.0.0.1:18080 \
  --token "$(cat runtime_data/sync_api_token.txt)" \
  sync markets/1604309 \
  --output-dir ./synced_records
```

### 5.4 同步并提交远端删除 ack

```bash
python local_sync_client/poly15_sync_client.py \
  --base-url http://127.0.0.1:18080 \
  --token "$(cat runtime_data/sync_api_token.txt)" \
  sync \
  --output-dir ./synced_records \
  --delete-remote
```

这一步的真实含义是：

- 先下载归档
- 先落本地 staging
- 校验 `size_bytes + sha256`
- 写本地 receipt 和 index
- 向服务端提交 ack

不会立即删远端文件。

### 5.5 只提交 ack，不立即同步

如果你已经同步过，想把当前 manifest 里的已确认 segments 标记为可删除：

```bash
python local_sync_client/poly15_sync_client.py \
  --base-url http://127.0.0.1:18080 \
  --token "$(cat runtime_data/sync_api_token.txt)" \
  delete --all \
  --output-dir ./synced_records
```

默认行为：

- 只会 ack 本地 `.poly15_sync_state/index.json` 里已确认的 segment
- 不会立即删除远端文件

强制模式：

```bash
python local_sync_client/poly15_sync_client.py \
  --base-url http://127.0.0.1:18080 \
  --token "$(cat runtime_data/sync_api_token.txt)" \
  delete --all \
  --output-dir ./synced_records \
  --force
```

`--force` 会绕过本地确认保护，只建议在你完全确认数据已安全落本地时使用。

### 5.6 手工触发服务端 GC

```bash
python local_sync_client/poly15_sync_client.py \
  --base-url http://127.0.0.1:18080 \
  --token "$(cat runtime_data/sync_api_token.txt)" \
  gc
```

作用：

- 清理所有 grace period 已到期、且 checksum 没变化的 acked segments

强制模式：

```bash
python local_sync_client/poly15_sync_client.py \
  --base-url http://127.0.0.1:18080 \
  --token "$(cat runtime_data/sync_api_token.txt)" \
  gc --force
```

`gc --force` 会忽略 grace deadline，直接尝试删除所有已 ack segment。

## 6. 推荐运行流程

推荐的日常运行顺序：

1. 启动 `collector`
2. 启动 `dashboard`
3. 定时执行本地 `sync`
4. 同步成功后执行 `sync --delete-remote` 或 `delete`
5. 等待后台 GC 自动删除，或者定时执行 `gc`

推荐思路：

- `collector` 和 `dashboard` 长期开着
- 本地同步客户端按 cron / 定时任务运行
- 服务端 GC 保持后台自动运行

## 7. 核对点

### 7.1 服务端是否在正常轮转

查看：

```bash
find runtime_data/records/sealed -type f | head
```

如果有输出，说明 `sealed segments` 已经开始产生。

### 7.2 是否已经有待删除的 ack

查看：

```bash
cat runtime_data/sync_state/gc_index.json
```

重点看每个 entry 的：

- `status`
- `acked_at`
- `delete_after`
- `last_error`

状态含义：

- `pending`：已经 ack，等待 GC
- `deleted`：已经被 GC 删除
- `stale`：GC 时发现文件内容变化，没有删除

### 7.3 本地同步是否真的落盘

查看：

```bash
find ./synced_records -type f | sort
```

重点看：

- 归档 segment 文件
- `.poly15_sync_state/index.json`
- `.poly15_sync_state/receipts/*.json`

## 8. 常见问题

### 8.1 为什么 `sync --delete-remote` 后 manifest 里文件还在？

因为现在是：

- 先 `ack`
- 再等待 grace period
- 最后由 GC 删除

所以这是正常行为。

### 8.2 为什么某些 acked 文件没有被删？

优先检查：

- `runtime_data/sync_state/gc_index.json`

如果 `status = stale`，通常表示：

- ack 之后远端文件又发生了变化
- checksum 或 size 不再匹配

这种情况下系统会拒绝删除，避免误删。

### 8.3 为什么同步不到 live 数据？

这是设计使然。sync API 只同步 `sealed segments`，不直接导出 live 文件，目的是避免“边写边删”。
